# R3 Chat + Profile + Overlays Audit

## Summary

14 files reviewed across Chat (4), Profile tab + 5 overlays (6), and secondary list pages (6). The web build is architecturally sound: all overlay routes wrap `<WebSlideOver open onClose={router.back()}>`, hooks are consumed correctly, and optimistic patterns are consistent. The critical gaps are: the event overlay's JOIN flow does not call `api.subscribeToEvent`, missing FR9.5 announcement composer in the web chat thread, the absence of block/report actions in the person profile overlay (FR11), the absence of the rate-event affordance in the event overlay (FR5.11), and an unconditional `SC_EVENTS` read in the interest detail overlay that fires in live mode. Several medium-severity wiring issues and a parity gap with the "failed message / retry" UI round out the findings.

---

## Findings

### F-R3-001: JOIN button in event overlay bypasses `api.subscribeToEvent` — no waitlist, no calendar, no RPC
- **Severity**: critical
- **File**: `scenecheck-expo/app/event/[id].web.tsx:66-75`
- **Category**: wiring-gap
- **Description**: `handleJoinToggle` calls only `joinEvent(id)` (store) and `leaveEvent(id)` (store). It never calls `api.subscribeToEvent` or `api.cancelSubscription`. This means: (1) subscriptions are never persisted in live mode; (2) waitlist logic (FR5.5) is never invoked; (3) no calendar side-effect can fire (FR7.2); (4) the "join group chat" toggle (FR7.3) is completely absent.
- **Evidence**:
  ```ts
  const handleJoinToggle = () => {
    if (!id) return;
    if (joined) {
      leaveEvent(id);
      showToast({ message: `Left "${event.title}".`, kind: 'info' });
    } else {
      joinEvent(id);
      showToast({ message: `Joined "${event.title}".`, kind: 'success' });
    }
  };
  ```
- **Mobile parity reference**: `scenecheck-expo/app/event/[id].tsx:178-278` — full async flow with `api.subscribeToEvent(id, true)`, waitlist toast, calendar insert, and rollback.
- **Spec reference**: FR5.5, FR7.2, FR7.3; Architecture §"Event subscription & calendar add"
- **Proposed fix**: Mirror the native `handleToggleJoin`. Replace the store-only calls with an `async` function that (a) optimistically calls `joinEvent`/`leaveEvent`, (b) calls `await api.subscribeToEvent(id, true)` or `await api.cancelSubscription(id)`, (c) checks `result.status === 'waitlisted'` and surfaces an appropriate toast, and (d) triggers `googleCalendar.insertEvent` when `linkedCalendar === 'google'` and the event has `startAt`. Rollback on API error.
- **Reasoning**: In live mode, joining without calling the RPC means the event_subscriptions row is never written, so the server has no record of this person's RSVP. The web calendar button already shows a "Calendar sync available on native" toast stub (line 199), which is correct for now, but the subscription itself must still persist.

### F-R3-002: No waitlist UI on the JOIN button (FR5.5 gap)
- **Severity**: critical
- **File**: `scenecheck-expo/app/event/[id].web.tsx:178-194`
- **Category**: wiring-gap
- **Description**: `WebJoinButton` is passed a static `label` of either "MANAGE EVENT" or "JOIN EVENT", with no capacity check. When the event is full, the button should read "JOIN WAITLIST" and the join flow should surface a waitlist-position toast.
- **Evidence**:
  ```ts
  <WebJoinButton
    joined={joined}
    onToggle={handleJoinToggle}
    size="lg"
    full
    label={event.kind === 'yours' ? 'MANAGE EVENT' : 'JOIN EVENT'}
  />
  ```
- **Mobile parity reference**: `scenecheck-expo/app/event/[id].tsx:562` — `label={!capUnknown && goingCount >= e.cap ? 'JOIN WAITLIST' : 'JOIN EVENT'}`
- **Spec reference**: FR5.5
- **Proposed fix**:
  ```ts
  const capUnknown = event.cap <= 0;
  const isFull = !capUnknown && goingCount >= event.cap;
  // Then pass:
  label={event.kind === 'yours' ? 'MANAGE EVENT' : (isFull ? 'JOIN WAITLIST' : 'JOIN EVENT')}
  ```
- **Reasoning**: FR5.5 requires the UI to distinguish a confirmed join from a waitlist add. Without this label change users cannot tell whether they are on the waitlist or attending.

### F-R3-003: FR9.5 announcement composer missing in `WebChatThread`
- **Severity**: high
- **File**: `scenecheck-expo/web/WebChatThread.tsx:467-530` (composer section)
- **Category**: parity-gap
- **Description**: The web composer is a plain textarea + send button with no announcement-mode toggle. The native chat thread checks `api.canPostAnnouncement(chatId)` and, when true, renders a toggle chip above the input. The web thread calls `send(text)` with no `messageType` argument (defaults to `'normal'`), so event hosts on web can never post an FR9.5 announcement. Receiving and rendering announcements is correctly wired (`isAnnouncement = m.messageType === 'announcement'`), but composing them is not.
- **Evidence** (composer, no announcement toggle):
  ```ts
  <textarea ... />
  <WebTip title="Send" side="top">
    <button type="button" onClick={submit} ...>
  ```
  `submit` calls `await send(text)` — no second argument.
- **Mobile parity reference**: `scenecheck-expo/app/chat/[id].tsx:62-76` — `api.canPostAnnouncement`, `setCanAnnounce`, `setAnnounceNext`.
- **Spec reference**: FR9.5
- **Proposed fix**: Add a `useEffect` that calls `api.canPostAnnouncement(chatId)` (same as native, respects the `cancelled` cleanup flag) and stores the result in `canAnnounce` state. When true, render a toggle chip above the composer. Pass `messageType: announceNext ? 'announcement' : 'normal'` to `send`.
- **Reasoning**: `useChatMessages.send` already accepts an optional `messageType` argument and forwards it through. The entire announcement infrastructure exists; it just needs to be surfaced in the web composer.

### F-R3-004: Block / Report absent from the person profile overlay (FR11)
- **Severity**: high
- **File**: `scenecheck-expo/app/profile/[id].web.tsx:84-97`
- **Category**: parity-gap
- **Description**: The web profile overlay renders only a "Report" button in the header row (line 86). That button fires a stub toast (`'Report submitted. Thanks.'`) via `useStore.getState().showToast` instead of calling `api.submitReport`. More critically, there is no "Block" action at all. The native screen (lines 313–358) has both `handleBlock` (calls `api.blockUser(id)`) and `handleReport` (calls `api.submitReport`) via a confirm dialog.
- **Evidence** (report stub, no block):
  ```ts
  <WebButton
    tone="ghost"
    size="sm"
    icon="flag"
    onClick={() =>
      useStore.getState().showToast({
        message: 'Report submitted. Thanks.',
        kind: 'info',
      })
    }
  >
    Report
  </WebButton>
  ```
- **Mobile parity reference**: `scenecheck-expo/app/profile/[id].tsx:313-358` — full block/report confirm dialogs calling `api.blockUser` / `api.submitReport`.
- **Spec reference**: FR11.1, FR11.2
- **Proposed fix**: For Report: replace the stub with `showConfirm({ ... onConfirm: () => api.submitReport(id, null, 'profile-report') })`. For Block: add a "Block" action (icon button or menu item) that calls `showConfirm` then `useStore.getState().blockUser(id, profile.name)` + `api.blockUser(id)` on confirm.
- **Reasoning**: Without a real API call, the report is silently discarded in live mode. The block is missing entirely, meaning a user on web has no safety recourse through this surface.

### F-R3-005: Rate-event affordance absent from event overlay (FR5.11)
- **Severity**: high
- **File**: `scenecheck-expo/app/event/[id].web.tsx` (entire file, no `canRate` guard)
- **Category**: parity-gap
- **Description**: The web event overlay does not implement the FR5.11 rating gate. The native screen computes `canRate = !isHost && isJoined && eventEnded` and conditionally renders a `RateEventSheet` trigger. The web overlay has no `endAt` computation, no `canRate` state, and no rate-event affordance.
- **Mobile parity reference**: `scenecheck-expo/app/event/[id].tsx:161-176, 543-557` — `endAt` derived from `e.startAt + e.endTime`, `canRate` condition, gold rate button.
- **Spec reference**: FR5.11
- **Proposed fix**: Port the `endAt` derivation (using `parseTime` from `lib/date-time`) and compute `const canRate = !isHost && joined && eventEnded`. Render a "Rate this event" button (visible only when `canRate === true`) that opens a `WebRateEventSheet` or navigates to a rate route. Even a minimal toast stub wired to a future sheet is better than nothing.
- **Reasoning**: FR5.11 is a graded requirement. The web build silently omits the rating affordance for attendees viewing past events.

### F-R3-006: `SC_EVENTS` read in live mode in `interests/[tag].web.tsx`
- **Severity**: high
- **File**: `scenecheck-expo/app/interests/[tag].web.tsx:50-52`
- **Category**: mock-leak
- **Description**: The "Events tagged #X" list is populated exclusively from `SC_EVENTS` (the mock fixture array), filtered client-side on `e.interests.includes(tag)`. There is no live-mode fallback. In production this will always show mock events.
- **Evidence**:
  ```ts
  const events = SC_EVENTS.filter(e =>
    (e.interests ?? []).includes(i.tag),
  );
  ```
- **Mobile parity reference**: `scenecheck-expo/app/interests/[tag].tsx` — uses `useInterest(tag)` which resolves live event data.
- **Spec reference**: AGENTS.md §Mock vs live mode
- **Proposed fix**: The comment on line 47 acknowledges a missing `/events?tag=` endpoint. The minimum acceptable fix: gate behind `api.isMock()`. When not in mock mode, either call a hook that fetches events by tag, or render an empty placeholder until the hook is wired up, rather than silently showing stale mock data.
- **Reasoning**: The AGENTS.md invariant states mock data must only appear inside a guarded mock-mode branch. Showing fixture events to a real user is misleading and violates that contract.

### F-R3-007: `SC_ME.interests` used unconditionally for shared-interests diff in `profile/[id].web.tsx`
- **Severity**: high
- **File**: `scenecheck-expo/app/profile/[id].web.tsx:65-66`
- **Category**: mock-leak
- **Description**: `const myInterests = SC_ME.interests ?? []` is read directly from the mock fixture at line 65, then used to compute `shared` interests at line 66. In live mode `SC_ME` is the mock default account (likely an empty or demo-populated object), so the shared-interest diff will always be empty or incorrect.

  ```ts
  const myInterests = SC_ME.interests ?? [];
  ```

  Alternatively, call `useUserInterests(meId)` and use its result.
- **Reasoning**: This is the same mock-leak class the AGENTS.md warns about. The diff label "2 shared" will be wrong in production.

### F-R3-008: `requests.web.tsx` — cancel outgoing request calls `api.removeFriend` instead of a cancel-request endpoint
- **Severity**: medium
- **File**: `scenecheck-expo/app/requests.web.tsx:64`
- **Category**: bug
- **Description**: The "Cancel" handler for outgoing pending requests calls `api.removeFriend(id)`. `api.removeFriend` deletes any friendship row matching `(me, other)` or `(other, me)`. For a pending request (status = `'pending'`) this is semantically wrong — the correct operation should be to delete only the pending friendship row or call a dedicated cancel-request method. The native sibling does the same thing (line 119 of `requests.tsx`), so this is a cross-platform bug. However, since `api.removeFriend` matches on either direction and pending requests are friendship rows, it will happen to work. Still, the intent mismatch is worth flagging: if a separate `api.cancelFriendRequest` is later added, this call site will be overlooked.
- **Evidence**:
  ```ts
  await api.removeFriend(id);
  ```
  Note `id` here is `p.id` — the other person's user ID (from `useOutgoingRequests`), not the friendship row id.
- **Proposed fix**: At minimum rename the variable to make the intent explicit in a comment. Ideally add `api.cancelFriendRequest(personId)` that wraps the same DELETE but is semantically named.
- **Reasoning**: Using a "remove friend" API to cancel a pending request is technically functional today but creates a maintenance trap. If the friendships schema or RLS is ever changed, the semantic mismatch will cause silent failures.

### F-R3-012: `WebChatList` mock data accessed without `isMock()` guard for `SC_EVENT_BY_ID`

### F-R3-014: `ReviewCard` in `ratings/[hostId].web.tsx` has a CSS `border` property conflict
- **Severity**: low
- **File**: `scenecheck-expo/app/ratings/[hostId].web.tsx:281-300`
- **Category**: bug
- **Description**: The "on {eventTitle}" footer button at line 281 sets both `paddingTop: 8` and `borderTop: \`1px solid ${t.line}\`` as a visual separator, but also sets `border: 'none'` at line 292 (the button reset). In CSS, `border: 'none'` is a shorthand that overrides `borderTop`. The separator line will never render.
- **Evidence**:
  ```ts
  style={{
    ...
    paddingTop: 8,
    borderTop: `1px solid ${t.line}`,   // <-- set first
    background: 'none',
    border: 'none',                       // <-- overrides borderTop
    ...
  }}
  ```
- **Proposed fix**: Move `border: 'none'` before `borderTop`, or split: use `borderTopWidth: 1, borderTopStyle: 'solid', borderTopColor: t.line` instead of the shorthand.
- **Reasoning**: The visual separator is intentional (mirrors the native event-title footer) but silently broken by the ordering of shorthand vs longhand CSS properties.

## Cross-cutting concerns

**Mock-data leaks across multiple files**: Three files in scope read SC_* fixtures in code paths that execute in live mode. The pattern is consistent — they all treat the fixture lookup as a "best-effort fallback" when live data is missing — but none is guarded by `api.isMock()` as the AGENTS.md invariant requires. Files affected: `WebChatList.tsx` (SC_EVENT_BY_ID), `WebChatThread.tsx` (SC_EVENT_BY_ID, SC_ACCOUNT_BY_ID), `interests/[tag].web.tsx` (SC_EVENTS unconditionally), `profile/[id].web.tsx` (SC_ME for shared-interests diff).

**Join/leave handler duplication**: The simple `joinEvent(id) + showToast` + `leaveEvent(id) + showToast` pattern is copy-pasted identically in `my-hosting.web.tsx`, `my-events.web.tsx`, `interests/[tag].web.tsx`, and `profile.web.tsx`. These all share the same limitation as F-R3-001 — in live mode they do not call `api.subscribeToEvent` / `api.cancelSubscription`. However, on list pages this is less critical because the primary join path is via the event overlay. Worth extracting into a shared `useJoinHandler()` hook to ensure any future live-mode wiring is applied in one place.

**`useStore.getState()` called inside a render callback**: In `profile/[id].web.tsx` line 90 the Report button's `onClick` calls `useStore.getState().showToast(...)`. This escapes the React subscription model and is a code smell — prefer capturing `showToast` via `const showToast = useStore(s => s.showToast)` at the component level, which is already the pattern in every other file in scope.

## Healthy / well-implemented patterns

- **Overlay routing is correctly wired throughout**: Every overlay file in scope (`event/[id].web.tsx`, `profile/[id].web.tsx`, `attendees/[id].web.tsx`, `ratings/[hostId].web.tsx`, `interests/[tag].web.tsx`) wraps content in `<WebSlideOver open onClose={() => router.back()}>` with `open` hardcoded to `true`, which is exactly the correct pattern for route-as-open-state. All five are registered with `presentation: overlayPres` (`transparentModal` on web, `card` on native) in `_layout.tsx`.

- **`useChatMessages` is consumed correctly**: `WebChatThread` destructures `send` directly from the hook and routes all sends through it. The optimistic pattern, Realtime subscription, and dedup logic are fully inherited without any new wiring.

- **Announcement rendering (receive path) is correct**: The web thread correctly handles `m.messageType === 'announcement'` for existing messages, rendering a tinted card with the bell icon and "ANNOUNCEMENT" label. This visually matches the native rendering pattern.

- **`requests.web.tsx` implements the full optimistic-commit pattern correctly**: `accept`, `decline`, and `cancel` all follow the AGENTS.md pattern — store update first, then API call, error toast on failure and `reload()` on success. This is the cleanest example of the pattern in the reviewed web files.
