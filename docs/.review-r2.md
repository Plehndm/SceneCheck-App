# R2 Home + Map + Search Audit

## Summary

The seven files in scope are cleanly structured, use the right hooks and store slices, and carry no direct fixture imports in the screens themselves. The wiring to `rank_events_query`, the optimistic join/leave flow, `wKindMeta` pin color-coding, and the `whenRange` time-range display are all correctly implemented. Seven findings follow — two high-severity (a mock-data leak in the shared `kind.ts` helper and a leave action on the Search screen that silently skips the API call), three medium, and two low.

---

## Findings

### F-R2-001: `wHostAccount` hard-returns mock fixtures in live mode (mock-data leak)
- **Severity**: high
- **File**: `scenecheck-expo/web/kind.ts:93`
- **Category**: mock-leak
- **Description**: `wHostAccount` unconditionally resolves the host by looking up `SC_ACCOUNT_BY_ID`, the static mock fixture dictionary. In live mode, `event.hostId` is a real UUID returned by `transformEventRow`; `SC_ACCOUNT_BY_ID` only knows the pre-seeded mock IDs (`me`, `p1`–`p6`, `org1`–`org3`). Any event hosted by a real Supabase user will have `hostId` that is absent from the dictionary, so `wHostAccount` returns `null` — the host row on every hover card and every `WebEventListCard` is silently blank in production.
- **Evidence**:
  ```ts
  export function wHostAccount(
    e: Pick<WebEvent, 'hostId'>,
  ): Account | null {
    if (e.hostId && SC_ACCOUNT_BY_ID[e.hostId]) return SC_ACCOUNT_BY_ID[e.hostId];
    return null;
  }
  ```
- **Spec reference**: FR4 / Architecture §"Map / event discovery" — host identity should render correctly for both user-created and scraped events.
- **Proposed fix**: The function needs to be async or the caller needs to pre-fetch host profiles. The cleanest minimal fix is to pass an optional `accountLookup` map (keyed by hostId) that each screen populates from a `getProfilesByIds` call after the event list arrives. In mock mode, seed the map from `SC_ACCOUNT_BY_ID`. This decouples the helper from the fixture dictionary:
  ```ts
  export function wHostAccount(
    e: Pick<WebEvent, 'hostId'>,
    lookup?: Record<string, Account>,
  ): Account | null {
    if (!e.hostId) return null;
    return lookup?.[e.hostId] ?? null;
  }
  ```
- **Reasoning**: `SC_ACCOUNT_BY_ID` is a mock fixture, and `kind.ts` is imported by `WebMap`, `WebEventListCard`, and `WebSearchAutocomplete` — all used in production. Every live-mode event whose host has not been pre-fetched silently shows an empty host row, which is a visible correctness gap on the hover card and list card.

### F-R2-002: Search screen `leaveEvent` skips `api.cancelSubscription` (missing API call)
- **Severity**: high
- **File**: `scenecheck-expo/app/search.web.tsx:108-111`
- **Category**: wiring-gap
- **Description**: The `onToggleJoin` handler in the web Discover/Search screen calls `leaveEvent(id)` (a store-only mutation) without calling `api.cancelSubscription`. Clicking "JOIN" (to join) is handled only by `joinEvent` — also store-only, no `api.subscribeToEvent`. The result is that joining or leaving an event from Search updates local state optimistically but never commits to the database in live mode. In contrast, the native `search.tsx` doesn't have join/leave at all (events open the detail screen on tap), and the Home and Map web screens both use the full `onJoin` pattern with `api.subscribeToEvent` / `api.cancelSubscription`.
- **Evidence**:
  ```ts
  const onToggleJoin = (id: string) => {
    if (joined.has(id)) leaveEvent(id);
    else joinEvent(id);
  };
  ```
- **Mobile parity reference**: `scenecheck-expo/app/search.tsx` — native search navigates to the event detail for join, so this divergence exists only on web.
- **Spec reference**: FR7.1 — subscription to an event must register the user as an attendee.
- **Proposed fix**: Replace `onToggleJoin` with the same optimistic-commit pattern used in `index.web.tsx:117–149` and `map.web.tsx:85–117`:
  ```ts
  const onToggleJoin = useCallback(async (id: string) => {
    const e = allEvents.find(x => x.id === id);
    if (!e) return;
    if (joined.has(id)) {
      leaveEvent(id);
      try { await api.cancelSubscription(id); }
      catch (err) { joinEvent(id); showToast({ ... }); }
    } else {
      joinEvent(id);
      try { await api.subscribeToEvent(id, true); }
      catch (err) { leaveEvent(id); showToast({ ... }); }
    }
  }, [allEvents, joined, joinEvent, leaveEvent, showToast]);
  ```
  Also requires pulling `showToast` and `useCallback` into the screen.
- **Reasoning**: The AGENTS.md invariant is "optimistic-commit pattern is universal: store update first, API call second, error toast + rollback on failure." The search screen violates this: the API call half is absent entirely.

### F-R2-003: `wKindMeta` falls back to `SC_ME.interests` when `subs` is omitted (silent mock fallback)
- **Severity**: medium
- **File**: `scenecheck-expo/web/kind.ts:38-40`
- **Category**: mock-leak
- **Description**: When `wKindMeta` is called without the `subs` argument (and `e.kind !== 'recommended'`), `wIsRecommended` builds a `Set` from `SC_ME.interests` — the hardcoded mock user's interest array. Every call to `wKindMeta(e, t)` without an explicit `subs` from the store's `subscribedInterests` therefore classifies events against the fixture user's interests, not the real user's. In the screens this is fine (`wKindMeta(event, t)` is called in `WebMap` and `WebEventListCard` which do not pass `subs`), but the "for you" count and the "RECOMMENDED" label will silently use mock interests in any live-mode scenario where the user has different subscriptions.
- **Evidence**:
  ```ts
  const set =
    subs instanceof Set
      ? subs
      : new Set(Array.isArray(subs) ? subs : SC_ME.interests || []);
  ```
- **Spec reference**: FR3.4 / FR4.1 — event-interest matching must reflect the signed-in user's actual subscriptions.
- **Proposed fix**: Require `subs` explicitly (or default to an empty Set, not `SC_ME.interests`) so callers are forced to pass the live store value. The screens already read `subscribedInterests` from the store for `wIsRecommended` calls in the `counts` memo; they should pass the same value to `wKindMeta` on individual cards too:
  ```ts
  // In kind.ts — change default to empty:
  : new Set(Array.isArray(subs) ? subs : []);
  // In WebMap/WebEventListCard — pass subscriptions:
  const { accent } = wKindMeta(e, t, subscribedInterests);
  ```
  `WebMap` already imports `useStore` so reading `subscribedInterests` there adds a single line.
- **Reasoning**: The fallback to `SC_ME.interests` is a mock fixture bleed. A live user with no overlapping interests with the mock data would see every event classified as "NEARBY" even if they do subscribe to matching tags, because `wKindMeta` inside the card used the mock set.

### F-R2-004: `transformEventRow` classifies non-creator non-scraped events as `'friend'` regardless of friendship
- **Severity**: medium
- **File**: `scenecheck-expo/lib/api.ts:122-125`
- **Category**: wiring-gap
- **Description**: The live-mode transform assigns `kind: 'friend'` to every event whose `creator_id` is not the current user and whose `source` is not `'scraped'` — a comment even acknowledges this is simplified. In practice, this means all user-created events hosted by strangers show the friend-colored pin and "FRIEND HOSTING" label on the web map/cards. The architecture document specifies pin color must distinguish User Created vs App Created (FR4.4), and the `wKindMeta` in `kind.ts` has a `'friend'` branch with a distinct accent. The web legend labels this bucket "Friends", so random strangers' events appear as friends' events.
- **Evidence**:
  ```ts
  const kind: SCEvent['kind'] = row.source === 'scraped'
    ? 'recommended'
    : row.creator_id === currentUserId
      ? 'yours'
      : 'friend'; // simplified — accurate version needs the friendship table
  ```
- **Mobile parity reference**: `scenecheck-expo/app/(tabs)/map.tsx` — native map uses the same `transformEventRow` path, so this is not a web-only gap, but the web legend makes the mislabeling more visible.
- **Spec reference**: FR4.4 — visual distinction between User Created Events and App Created Events; the legend in `WebMap` shows "Friends" as one bucket, so the "friend" kind should only apply to actual friends.
- **Proposed fix**: The fix requires the `rank_events_query` RPC to return a `is_friend_creator` boolean (Supabase can join `friendships` in the RPC). Until then, a safe interim is to classify non-creator non-scraped events as `'other'` rather than `'friend'`, and only escalate to `'friend'` when a server-supplied flag says so. This is the correct conservative default.
- **Reasoning**: Showing strangers' events as "FRIEND HOSTING" with the friend-colored pin is a correctness and UX bug. The legend is a promise to the user; breaking it erodes trust.

### F-R2-005: Web Map has no location-denied/unavailable affordance
- **Severity**: medium
- **File**: `scenecheck-expo/app/(tabs)/map.web.tsx` (entire file)
- **Category**: parity-gap
- **Description**: The native `map.tsx` renders an explicit "TAP TO ENABLE LOCATION" / "LOCATION DENIED · USING UCI DEFAULT" button when `isFallback` is true (lines 107-122), calling `request()` from `useLocation`. The web `map.web.tsx` never reads `status`, `isFallback`, or `request` — it silently falls back to the UCI default with no UI affordance. On web, browsers block automatic geolocation prompts (the `useLocation` hook sets status to `'idle'` on web and never calls `request()` on mount). The user therefore gets the UCI fallback with no way to trigger a permission request from the Map tab.
- **Evidence** (web map, no fallback UI):
  ```ts
  const { coords } = useLocation();
  // `status`, `isFallback`, `request` are never destructured
  ```
- **Mobile parity reference**: `scenecheck-expo/app/(tabs)/map.tsx:107-122`
- **Spec reference**: FR1.5 — "App requests location permission after sign-up and on first map view." FR4.1 — the map must display nearby events filtered by the user's actual location.
- **Proposed fix**: Destructure `{ coords, status, isFallback, request }` from `useLocation()` and render a small floating pill (similar to the native version) above the map when `isFallback` is true:
  ```tsx
  {isFallback && (
    <div style={{ position: 'absolute', top: 72, left: '50%', transform: 'translateX(-50%)', zIndex: 50 }}>
      <button onClick={request} style={...}>
        {status === 'denied' ? 'LOCATION DENIED · SHOWING UCI DEFAULT' : 'ENABLE LOCATION'}
      </button>
    </div>
  )}
  ```
- **Reasoning**: Web geolocation requires a user gesture to trigger the browser permission dialog. Without the affordance, users who have not yet granted location can only see events near UCI — silently, with no explanation or way to correct it.

### F-R2-006: `WebEventListCard` and `WebMapHoverCard` omit price chip (parity gap with native card)
- **Severity**: low
- **File**: `scenecheck-expo/web/WebEventListCard.tsx`, `scenecheck-expo/web/WebMap.tsx:680-693`
- **Category**: parity-gap
- **Description**: The native `SCEventCard` renders a `PriceChip` (a "FREE" badge or dollar amount) on each card's bottom bar, sourced from `formatPrice` / `priceState` in `lib/price.ts`. Neither `WebEventListCard` nor `WebMapHoverCard` import or render price information. The `WebCapBar` (capacity bar) is correctly present in both, but the FREE / paid distinction that `SCEventCard` surfaces at a glance is absent from all web cards.
- **Mobile parity reference**: `scenecheck-expo/components/SCEventCard.tsx:98-109, 117-140`
- **Spec reference**: FR4.5 — display of participant count (done via `WebCapBar`) is present; price surfacing is an adjacent discovery-quality feature.
- **Proposed fix**: Import `formatPrice` and `priceState` from `@/lib/price` and add a `PriceChip`-style inline element to `WebEventListCard` alongside the capacity bar:
  ```tsx
  import { formatPrice, priceState } from '@/lib/price';
  // Inside the bottom row:
  {priceState(event) !== 'none' && (
    <span style={{ fontFamily: FONT.mono, fontSize: 10, fontWeight: 700,
      color: priceState(event) === 'free' ? t.good : t.ink2 }}>
      {formatPrice(event)}
    </span>
  )}
  ```
- **Reasoning**: Users rely on the "FREE" badge to quickly identify events that don't require a ticket. Omitting it from the web cards creates a discovery friction gap relative to the native experience, and events that do carry a price show no hint until the detail page.

### F-R2-007: `WebDiscSection` empty-state logic does not distinguish "loading" from "no results"
- **Severity**: low
- **File**: `scenecheck-expo/web/WebDiscSection.tsx:34-35`, `scenecheck-expo/app/search.web.tsx:233-238`
- **Category**: parity-gap / bug
- **Description**: `WebDiscSection` treats `count === 0` as the empty state and renders `emptyText` immediately. The calling code passes `events.length` as `count`, which is `0` while `useEvents` is still loading. The `emptyText` for the events section is `eventsLoading ? 'Loading events…' : 'No matching events nearby.'` — a conditional string — but `WebDiscSection` shows it the moment `count` becomes `0`, so on first load the user briefly sees "Loading events…" as the section body text before data arrives. There is no spinner or skeleton in the section. The native `search.tsx` uses `SCListSkeleton` during the loading phase (line 234).
- **Evidence**:
  ```tsx
  <WebDiscSection
    title="Events"
    count={events.length}
    emptyText={eventsLoading ? 'Loading events…' : 'No matching events nearby.'}
  >
  ```
- **Mobile parity reference**: `scenecheck-expo/app/search.tsx:234` — uses `SCListSkeleton rows={5}` during loading.
- **Proposed fix**: Pass a `loading` prop to `WebDiscSection` (or handle it at the caller level) and render a skeletal placeholder instead of `emptyText` when loading is true and count is zero. Alternatively, pass `count={undefined}` while loading so the section doesn't enter the empty-state branch until the first response lands.
- **Reasoning**: Showing "Loading events…" as body text is a low-polish experience compared to a skeleton loader and inconsistent with the native screen. It also makes the section header show "Events 0" during loading, which is confusing.

## Cross-cutting concerns

**`FiltersPopover` is duplicated verbatim** between `index.web.tsx:521-704` and `map.web.tsx:225-408`. The code comment in `map.web.tsx` acknowledges this ("Mirror of the popover in index.web.tsx — kept local rather than hoisted"). Both copies are currently identical — any future change (e.g. adding a "scraped only" toggle, fixing a label) must be applied to both. This is a maintainability risk rather than a functional bug today, but deserves a shared `WebMapFilterPopover` component when a third caller appears.

**`wKindMeta` / `wIsRecommended` `subs` contract is implicit.** Three callers pass `subscribedInterests` explicitly (the `counts` memos in `index.web.tsx` and `map.web.tsx`) but two critical per-card calls (`wKindMeta(e, t)` inside `WebMap`'s pin loop and `WebEventListCard`'s render) omit `subs` entirely, silently using `SC_ME.interests` (finding F-R2-003). The function signature makes `subs` optional which invites this mistake. Making it required (or defaulting to an empty Set) would surface missing call-sites at compile time.

## Healthy / well-implemented patterns

- **Optimistic join/leave on Home and Map** (`index.web.tsx:117-149`, `map.web.tsx:85-117`) correctly implements the AGENTS.md invariant: store mutation first, API call second, error toast + rollback. The 5-second undo-leave grace period with a store `pendingLeave` Set is a well-thought-out UX pattern.
- **`useEvents` radius wiring** threads `radiusM` (derived from the persisted `useStore.radius` in miles) into the `rank_events_query` RPC call, so changing the distance chip in the filter popover immediately writes back to the persisted store and refetches from PostGIS on both the Home and Map tabs — exactly matching the architecture document's FR4.2 specification.
- **`WebSearchAutocomplete` scoring algorithm** correctly ports the prototype's five-tier priority (title prefix → title contains → venue → interest tag → host name) with a `MAX_SUGGESTIONS = 6` bound and a graceful `onSearch` fallback when no suggestion matches, routing to the full Discover screen.
- **`useLocation` web-mode awareness** — `index.web.tsx` and `map.web.tsx` both pass `coords` explicitly to `useEvents` rather than relying on an implicit default, and they read `coords?.latitude` (optional chaining) defensively, so the map and event list render correctly while location is `'idle'` or `'denied'`.
