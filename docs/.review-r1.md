# R1 Foundation Audit

## Summary

The web foundation slice is architecturally sound: the shell/rail/atom pattern is clean, badge wiring is live (not hardcoded), and the platform split in `_layout.tsx` is correctly guarded. I found **2 critical**, **4 high**, **5 medium**, and **3 low** findings. The critical issues are a persistent mock-data leak in `WebFriendButton` and `WebFollowButton` that fires in live sessions, and a focus-trap absence on the activity panel that breaks keyboard navigation.

---

## Findings

### F-R1-001: Mock leak â€” `SC_ACCOUNT_BY_ID` queried unconditionally in WebFriendButton (live mode)

- **Severity**: critical
- **File**: `scenecheck-expo/web/WebFriendButton.tsx:34`
- **Category**: mock-leak
- **Description**: The component imports `SC_ACCOUNT_BY_ID` from `@/data/mocks` and looks up `personId` on every render without any `api.isMock()` guard. In live mode `personId` is a real UUID, so `SC_ACCOUNT_BY_ID[personId]` returns `undefined`. The `p?.privacy` check then silently evaluates to `'public'` (the else branch), so every live user is treated as a non-private account â€” meaning the button always shows "Add friend" and never "Request" even for private profiles. Additionally the `removeFriend` and `sendReq` toast messages fall back to `p?.name || 'friend'`, which always hit the fallback in live mode.
- **Evidence**:
  ```ts
  import { SC_ACCOUNT_BY_ID } from '@/data/mocks';
  // ...
  const p = SC_ACCOUNT_BY_ID[personId] ?? null;
  // ...
  label: p?.privacy === 'private' ? 'Request' : 'Add friend',
  ```
- **Spec reference**: FR8.1 (friend requests), FR8.2 (private account approval flow)
- **Mobile parity reference**: `scenecheck-expo/components/AuthBootstrap.tsx` hydrates profile data into the store; the web button should read `privacy` from an `api.getProfile()` call or a cached store value, not from a mock fixture.
- **Proposed fix**: The component already has access to the store's `friends` / `outgoingRequests` sets for status â€” add a `useProfile(personId)` hook (or a local `useEffect` that calls `api.getProfile`) to retrieve the live `privacy` field. In mock mode, `api.getProfile` already returns `SC_ACCOUNT_BY_ID`, so the mock branch is preserved transparently. Remove the `SC_ACCOUNT_BY_ID` import from this file.
- **Reasoning**: The AGENTS.md invariant states mock data is only acceptable when `api.isMock()` is true or no session exists. This import bypasses that gate entirely.

### F-R1-002: Mock leak â€” `SC_ACCOUNT_BY_ID` queried unconditionally in WebFollowButton (live mode)

- **Severity**: critical
- **File**: `scenecheck-expo/web/WebFollowButton.tsx:29`
- **Category**: mock-leak
- **Description**: Same root cause as F-R1-001. `SC_ACCOUNT_BY_ID[orgId]` is always `null` in live mode (real UUID key), so `o?.privacy` is always falsy, and every org is presented as public ("Follow" label, never "Request to follow"). The `toggleFollow` store action also calls `api.followOrg` / `api.unfollowOrg` (via the store's `toggleFollow`), but the store's `toggleFollow` mutator does not call the API â€” it only flips the local Set. So the follow action is never written to the `org_follows` table in live mode.
- **Evidence**:
  ```ts
  import { SC_ACCOUNT_BY_ID } from '@/data/mocks';
  // ...
  const o = SC_ACCOUNT_BY_ID[orgId] ?? null;
  // ...
  label: o?.privacy === 'private' ? 'Request to follow' : 'Follow',
  ```
- **Spec reference**: FR7.1 / FR7.2 (org follow, private org request flow)
- **Proposed fix**: Same as F-R1-001 for the privacy label. For the API wiring gap: `toggleFollow` in `useStore.ts` only mutates the local Set and never calls `api.followOrg` / `api.unfollowOrg`. `WebFollowButton` should call those API methods directly after the optimistic store update (consistent with the project's optimistic-commit pattern from AGENTS.md). Example:
  ```ts
  const onClick = async (e: MouseEvent<HTMLButtonElement>) => {
    e.stopPropagation();
    toggle(orgId); // optimistic
    try {
      if (status === 'following') await api.unfollowOrg(orgId);
      else await api.followOrg(orgId);
    } catch {
      toggle(orgId); // rollback
      showToast({ message: 'Failed', kind: 'error' });
    }
    // ... toast
  };
  ```
- **Reasoning**: AGENTS.md optimistic-commit pattern: store update first, API call second, rollback on failure. The current code has no API call at all.

### F-R1-003: Activity panel missing focus trap â€” keyboard users can tab behind the modal

- **Severity**: high
- **File**: `scenecheck-expo/web/WebActivityPanel.tsx:49`
- **Category**: a11y
- **Description**: The panel has `role="dialog"` and `aria-modal="true"` but no focus trap. When the panel opens, focus is not moved into the dialog and tab order continues behind the backdrop into the main content area. Screen readers interpret `aria-modal` as a hint that focus should stay inside, but browsers do not enforce this automatically â€” a focus trap is required. ESC closes the panel correctly (line 43), but focus is not returned to the bell button that opened it.
- **Evidence**:
  ```tsx
  <div
    role="dialog"
    aria-modal="true"
    aria-label="Activity"
    style={{ position: 'absolute', top: 12, left: 12, ... }}
  >
  ```
  No `autoFocus`, no `tabIndex`, no focus-trap library call, no `onClose` restoring focus to trigger.
- **Mobile parity reference**: Native modal stacks (`presentation: 'transparentModal'`) enforce focus containment via the OS. The web panel is a DIV overlay that must replicate this manually.
- **Spec reference**: FR10 (notifications â€” users must be able to interact with and dismiss notifications).
- **Proposed fix**: On open, move focus to the first focusable child (or the close button) and restore it to the rail bell button on close. A minimal implementation:
  ```ts
  const panelRef = useRef<HTMLDivElement>(null);
  const triggerRef = useRef<HTMLButtonElement>(null); // passed from rail
  useEffect(() => {
    if (open) panelRef.current?.querySelector<HTMLElement>('[autofocus],[tabindex="0"],button')?.focus();
    else triggerRef.current?.focus();
  }, [open]);
  ```
  Alternatively, constrain tab with a keydown handler on the panel that wraps focus within focusable children.
- **Reasoning**: WCAG 2.1 SC 2.1.2 (No Keyboard Trap) and SC 2.4.3 (Focus Order) â€” modal dialogs must contain focus. Users who navigate entirely by keyboard cannot close or interact with the panel without this.

### F-R1-004: `WebSlideOver` missing focus trap and aria-label

- **Severity**: high
- **File**: `scenecheck-expo/web/WebSlideOver.tsx:74`
- **Category**: a11y
- **Description**: `WebSlideOver` has `role="dialog"` and `aria-modal="true"` but, like the activity panel, has no focus trap and no `aria-label` or `aria-labelledby`. Every slide-over (event detail, profile, attendees, ratings, interests) inherits this gap. Without `aria-label`, assistive technology announces the dialog as "dialog" with no description.
- **Evidence**:
  ```tsx
  <div
    role="dialog"
    aria-modal="true"
    style={{ position: 'absolute', top: 0, right: 0, ... }}
  >
    {children}
  </div>
  ```
- **Proposed fix**: Add an `aria-label` prop to `WebSlideOver` (passed from each consumer, e.g. `aria-label="Event details"`) and apply the same focus-management pattern described in F-R1-003. The close button rendered by each consumer screen should receive `autoFocus` or the panel should programmatically focus its first interactive element on open.
- **Reasoning**: Same WCAG requirements as F-R1-003. All overlay routes use `WebSlideOver` on web, so this affects every detail screen a user navigates to.

### F-R1-005: `wHostAccount` in `kind.ts` is an unconditional mock leak â€” always returns `null` in live mode

- **Severity**: high
- **File**: `scenecheck-expo/web/kind.ts:93`
- **Category**: mock-leak
- **Description**: `wHostAccount` looks up the host in `SC_ACCOUNT_BY_ID` unconditionally. In live mode the `hostId` on an event is a mock short-ID or a UUID (depending on which path created the event), but `SC_ACCOUNT_BY_ID` only has mock fixture keys. Any screen consuming `wHostAccount` in live mode always gets `null`, meaning host attribution on event cards (name, avatar, org label) is blank for all live events.
- **Evidence**:
  ```ts
  export function wHostAccount(
    e: Pick<WebEvent, 'hostId'>,
  ): Account | null {
    if (e.hostId && SC_ACCOUNT_BY_ID[e.hostId]) return SC_ACCOUNT_BY_ID[e.hostId];
    return null;
  }
  ```
- **Spec reference**: FR4.4 (event kind distinction requires knowing the host type â€” person vs org).
- **Proposed fix**: `wHostAccount` is a synchronous helper but resolving a live profile is async. The right fix is to move host resolution into the calling component via a `useProfile(hostId)` hook (which calls `api.getProfile` in live mode and falls back to `SC_ACCOUNT_BY_ID` in mock mode). The `wHostAccount` export should be deprecated or given an `api.isMock()` guard that returns `null` in live mode with a comment pointing to the hook.
- **Reasoning**: The function is called on event cards across the web UI; silently returning `null` in live mode gives every event an anonymous host rather than a visible error, making the bug hard to notice during development.

### F-R1-006: `wIsRecommended` falls back to `SC_ME.interests` in live mode when `subs` is not passed

- **Severity**: high
- **File**: `scenecheck-expo/web/kind.ts:38`
- **Category**: mock-leak
- **Description**: When `subs` is omitted (falsy), the function constructs a Set from `SC_ME.interests` â€” the hardcoded mock user's interests â€” rather than the signed-in user's live subscriptions. Any event card or map pin that calls `wKindMeta` without explicitly passing `subscribedInterests` from the store will classify events against the mock user's interests (`['biking', 'coffee', 'climbing']`) instead of the real user's choices.
- **Evidence**:
  ```ts
  const set =
    subs instanceof Set
      ? subs
      : new Set(Array.isArray(subs) ? subs : SC_ME.interests || []);
  ```
- **Proposed fix**: The fallback should be an empty Set, not `SC_ME.interests`:
  ```ts
  : new Set<string>();
  ```
  All callers that care about recommendations should pass `subscribedInterests` from `useStore(s => s.subscribedInterests)` explicitly. This makes the dependency visible and avoids silent mock contamination in live builds.
- **Reasoning**: In live mode a new user with no interests (or different interests) would see recommendation labels driven by a hardcoded mock persona. This directly violates the AGENTS.md invariant that mock-data only flows when `api.isMock()` is true.

### F-R1-007: Account switcher popover has no keyboard dismissal and no ARIA role

- **Severity**: medium
- **File**: `scenecheck-expo/web/WebRail.tsx:636`
- **Category**: a11y
- **Description**: The account-switcher popover opens on button click but has no `role` (should be `menu` or `listbox`), its items have no `role="menuitem"`, there is no ESC handler to close it (ESC only closes the activity panel, not this popover), and clicking outside the rail does close it via `onMouseLeave` but that does not work for keyboard users who tabbed into the popover.
- **Evidence**:
  ```tsx
  {switcherOpen && (
    <div
      style={{
        position: 'absolute',
        bottom: 56,
        ...
      }}
    >
  ```
  No `role`, no ESC handler, no `aria-expanded` link between trigger button and popover.
- **Proposed fix**: Add `role="menu"` to the popover `div`, `role="menuitem"` to each account button, set `aria-expanded={switcherOpen}` on the trigger button, and add a `useEffect`-based ESC handler (same pattern as `WebActivityPanel`). Consider using `aria-haspopup="menu"` on the trigger.
- **Reasoning**: Keyboard users navigating the rail with Tab can open the switcher (it is a button) but cannot close it with ESC or navigate its items with arrow keys.

### F-R1-008: Rail badge for `requests` uses `incomingRequests.size` which is a live Set â€” but the `badge` prop rendering clamps to positive values only via `|| null`; zero-count is fine, but `size` is fine too â€” the real issue is no optimistic-decrement

- **Severity**: medium
- **File**: `scenecheck-expo/web/WebShell.tsx:69`
- **Category**: parity-gap
- **Description**: `badges.requests` is set from `incoming.size || null` where `incoming` is `useStore(s => s.incomingRequests)`. The `incomingRequests` Set is updated by `acceptFriendRequest` and `declineFriendRequest` store mutators, and `WebActivityRow` calls those mutators correctly. However the rail badge is not recalculated until the next Zustand selector re-render, which is immediate â€” this part is fine. The parity gap is that `useNotifications` is called twice: once in `WebShell` (line 58, for the badge) and once in `WebActivityPanel` (line 38, for the body). The singleton pattern in `useNotifications` handles this correctly (no double subscription), but the `loading` and `error` state are read from module-level variables via closure, and both callers receive the same stale snapshot until `emit()` fires. This is by design but the `error` state written into `markRead`/`markAllRead` failures (line 200, line 218 in `useNotifications.ts`) mutates `_initialError` at the module level â€” that error will appear in the badge component's `{ error }` destructure even though the badge never displays errors. Not user-visible, but the badge component receives an unexposed error reference.
- **Evidence**:
  ```ts
  // WebShell.tsx:58
  const { data: notifData } = useNotifications();
  ```
  The badge only uses `data`, but `useNotifications` always provides `error` at module scope which can be non-null after a `markRead` failure in `WebActivityPanel`.
- **Proposed fix**: Not urgent. Document in `useNotifications.ts` that `_initialError` conflates load errors and mutation errors, or split them into separate module-level variables so load-time failures and operation failures are separately queryable.
- **Reasoning**: Medium because it is not user-visible but it can cause confusing state in tests that check for error conditions.

### F-R1-009: `WebTag` renders as a `<button>` even when `onClick` is not provided â€” not semantic

- **Severity**: medium
- **File**: `scenecheck-expo/web/WebTag.tsx:41`
- **Category**: a11y
- **Description**: When `onClick` is undefined the element is non-interactive (cursor: default) but still rendered as a `<button type="button">`. Screen readers announce it as a button, users can tab to it, and pressing Enter/Space fires a no-op click. For purely decorative tags (no action) this is misleading. The native sibling `SCTag.tsx` renders a `View`/`Text` for non-interactive tags and a `TouchableOpacity` for interactive ones.
- **Evidence**:
  ```tsx
  <button
    type="button"
    onClick={onClick}
    style={{
      cursor: onClick ? 'pointer' : 'default',
    ```
- **Mobile parity reference**: `scenecheck-expo/components/SCTag.tsx` â€” non-interactive tags use a plain `View`, interactive ones use a `Pressable`.
- **Proposed fix**:
  ```tsx
  if (!onClick) {
    return <span style={...}>...</span>;
  }
  return <button type="button" onClick={onClick} style={...}>...</button>;
  ```
- **Reasoning**: The pattern of conditionally rendering a semantic element based on interactivity is standard and matches the native sibling.

### F-R1-010: `WebDiscSection` hides `children` when `count === 0` even if children were passed

- **Severity**: medium
- **File**: `scenecheck-expo/web/WebDiscSection.tsx:35`
- **Category**: bug
- **Description**: `isEmpty` is derived from `count === 0`, not from `children == null`. If the parent passes `count={0}` but also passes children (e.g. an "empty" custom illustration component), those children are suppressed. More subtly: if `count` is `undefined` (not passed), `isEmpty` is `false` and children render â€” but if the caller forgets `count` when they do have zero results, children render incorrectly.
- **Evidence**:
  ```ts
  const isEmpty = count === 0;
  // ...
  {isEmpty ? (
    <div>...{emptyText}</div>
  ) : (
    children
  )}
  ```
- **Proposed fix**: Use `children == null` (no children passed at all) as the secondary gate, or change the condition to `isEmpty && !children` so passed children always win:
  ```tsx
  {isEmpty && !children ? (
    <div style={{ color: t.ink3, ... }}>{emptyText}</div>
  ) : (
    children
  )}
  ```
- **Reasoning**: The component is a shared wrapper used across the search screen. If a screen passes count=0 and a skeleton/empty-state component as children, those are swallowed and only the generic text shows.

### F-R1-011: `WebSettingsSection` uses `9999` as the unconstrained `max-height` fallback before ResizeObserver measures

- **Severity**: low
- **File**: `scenecheck-expo/web/WebSettingsSection.tsx:168`
- **Category**: bug
- **Description**: Before `ResizeObserver` fires its first callback, `bodyHeight` is `null` and the `maxHeight` expression evaluates to `9999 + 32 = 10031px`. On first open the section animates from `0 â†’ 10031px`, which causes a visible "snap" because the actual content may be only ~200px tall â€” the transition runs to 10031px and then the layout collapses. Only affects the first expand before RO fires.
- **Evidence**:
  ```tsx
  maxHeight: open ? (bodyHeight ?? 9999) + 32 : 0,
  ```
- **Proposed fix**: Use a two-phase open: animate to `10031px` is fine if you also accept the snap. A cleaner approach:
  ```tsx
  // Start with height: 'auto' if bodyHeight hasn't been measured yet,
  // only animate once RO has a real value.
  maxHeight: open ? (bodyHeight != null ? bodyHeight + 32 : 'none') : 0,
  ```
  Or initialise `bodyHeight` to `0` and rely on RO firing synchronously on first mount (which browsers guarantee for already-rendered elements).
- **Reasoning**: Low impact â€” the snap happens only on the very first expand, before RO fires. Not a correctness bug, just a UX jitter.

### F-R1-013: `WebAvatar` initials can produce an empty string for names with no spaces (single-word names)

- **Severity**: low
- **File**: `scenecheck-expo/web/WebAvatar.tsx:36`
- **Category**: bug
- **Description**: `initials` is built by splitting on spaces and taking the first char of each word. For a single-word name like `"Anthropic"` the result is `"A"` (correct). For a name like `"?"` (the fallback) or an empty string, the result is `""` or `"?"` respectively. The issue is the fallback for `person?.name || '?'`: when `name` is an empty string (`''`), JS coerces it to falsy, so the fallback `'?'` is used â€” which renders a `?` initial on the avatar for any user whose `name` column is empty in the DB. This is consistent but slightly odd UX; the native `SCAvatar` uses the same empty-string fallback idiom. Not a crash but a cosmetic gap worth tracking.
- **Evidence**:
  ```ts
  const initials = (person?.name || '?')
    .split(' ')
    .map(w => w[0])
    .slice(0, 2)
    .join('');
  ```
- **Proposed fix**: Render nothing or the user icon when name is truly empty, e.g. `(person?.name?.trim() || '?')`. Very low priority.
- **Reasoning**: In practice the DB enforces a non-empty name via the `handle_new_user` trigger, but the guard is worth noting for defensive rendering.

## Cross-cutting concerns

**Mock imports in live-session components (F-R1-001, F-R1-002, F-R1-005, F-R1-006)**: Four files in the foundation slice import from `@/data/mocks` without an `api.isMock()` guard. This is the single most impactful issue in the slice: it means friend buttons, follow buttons, host attribution, and event recommendations all silently degrade in live mode. The fix pattern is consistent: replace `SC_ACCOUNT_BY_ID[id]` calls with `api.getProfile(id)` (which already has its own mock branch), and replace the `SC_ME.interests` fallback with an empty Set. Other agents (R2â€“R4) should check their screens for the same pattern â€” any call to `SC_*` constants outside an `if (api.isMock())` block is a live-mode data gap.

**Focus management on overlay surfaces (F-R1-003, F-R1-004, F-R1-007)**: Three overlay surfaces lack focus traps or focus restoration. This is a systematic gap in the web foundation â€” the shell was built mobile-first and keyboard navigation was not part of the initial pass. The pattern to apply is: on open, query the panel for focusable children and call `.focus()` on the first; on close, return focus to the trigger element. A shared `useFocusTrap(open, panelRef, triggerRef)` hook would address all three sites and any future overlays.

## Healthy / well-implemented patterns

- **Live badge wiring in WebShell**: The rail's notification, chat, and friend-request badges are all derived from real hooks (`useNotifications`, `useChats`, `useStore.incomingRequests`) and will update in real-time via Supabase Realtime subscriptions. No hardcoded zeros.
- **`useNotifications` singleton with refcount teardown**: The module-level singleton + listener fan-out pattern is the correct solution for multiple simultaneous consumers of the same Realtime channel. The refcount teardown prevents channel leaks when both the badge and the panel are mounted simultaneously.
- **Platform split in `_layout.tsx`**: The `require('@/web/WebShell').WebShell` pattern correctly excludes the DOM-dependent module from the native bundle without using platform-extension files for a component that is purely layout-level.
- **`WebSettingsSection` ResizeObserver + animated max-height**: The ResizeObserver approach for smooth collapse animation when the section body resizes (e.g. discovery-radius warning callout) is a good pattern â€” it avoids hard-coding `max-height` values that break when content changes.
