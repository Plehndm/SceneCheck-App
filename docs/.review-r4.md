# R4 Create + Settings + Notifications Audit

## Summary

9 findings across 7 files. The slice is well-structured: settings.web.tsx correctly
reads every store slice the native sibling reads, no mock data leaks appear in settings,
and useNotifications implements a solid module-singleton with correct refcount teardown.
The main concerns are (1) `visibility` is captured by the wizard form but silently
dropped from the `api.createEvent` call body, (2) `autoGroupChat` is displayed and
stored in the draft but never forwarded to the Edge Function, (3) the `SC_MY_ACCOUNTS`
import in create-event.web.tsx is an unguarded mock-data reference used in live mode,
and (4) a boolean-precedence bug in the draft step-restore expression that will always
land the user on step 1 when resuming.

## Findings

### F-R4-001: `visibility` silently dropped from createEvent payload
- **Severity**: high
- **File**: `scenecheck-expo/app/create-event.web.tsx:239-255`
- **Category**: wiring-gap
- **Description**: The wizard collects `form.visibility` in Step 3 and shows it in the
  Review step, but it is never included in the object passed to `api.createEvent`. The
  architecture doc's RPC signature (`POST /rest/v1/rpc/create_event`) does not list
  `visibility` as a required field, but the `DraftForm` type carries it and the native
  `create-event.tsx` is expected to pass it through. Without it every event created on
  web is implicitly public regardless of the host's choice.
- **Evidence**:
  ```ts
  const result = await api.createEvent({
    title: form.title.trim(),
    description: form.desc.trim(),
    start_at: friendlyToISO(form.date, form.timeStart),
    end_at: friendlyToISO(form.date, form.timeEnd),
    location: { lat: form.lat ?? coords.latitude, lng: form.lng ?? coords.longitude },
    location_name: form.location.trim(),
    capacity: form.cap,
    min_subscribers: form.minSubs,
    interests: form.interests,
    price_min: priceMin,
    price_max: priceMax,
    price_currency: priceCurrency,
    // ← visibility: form.visibility is absent
  });
  ```
- **Spec reference**: FR5.1 (event visibility is a creator-set field); Architecture
  §"User-created events (FR5)"
- **Proposed fix**: Add `visibility: form.visibility` to the createEvent call body, and
  confirm the `create-event` Edge Function accepts + stores the field.
- **Reasoning**: A private event that silently publishes publicly is a privacy defect.
  The user made a choice; the system must honor it.

### F-R4-002: `autoGroupChat` collected but never forwarded to Edge Function
- **Severity**: high
- **File**: `scenecheck-expo/app/create-event.web.tsx:239-255`
- **Category**: wiring-gap
- **Description**: The wizard's Step 3 exposes an "Auto-create group chat" toggle wired
  to `form.autoGroupChat`, and the Review step reflects the choice. However the
  `handlePublish` call to `api.createEvent` does not include `auto_group_chat` (or any
  equivalent key) in its payload. FR9.4 requires that the system auto-create a group
  chat for each event; this toggle allows the creator to opt out. Without the flag
  reaching the Edge Function, the server cannot respect the opt-out.
- **Evidence**:
  ```ts
  // Step 3 toggle writes form.autoGroupChat = true/false
  <WebToggleRow
    label="Auto-create group chat"
    value={form.autoGroupChat}
    onChange={(v) => onChange({ autoGroupChat: v })}
  />
  // handlePublish never includes auto_group_chat
  const result = await api.createEvent({
    title: ..., description: ..., // ... no autoGroupChat field
  });
  ```
- **Spec reference**: FR9.4 (auto group chat on event creation)
- **Proposed fix**: Pass `auto_group_chat: form.autoGroupChat` in the createEvent call
  body; confirm the Edge Function reads and stores it.
- **Reasoning**: If the user turns this off, the server still auto-creates a chat, making
  the toggle a no-op. That is confusing and violates the spec's opt-in model.

### F-R4-003: Unguarded `SC_MY_ACCOUNTS` import used in live mode
- **Severity**: high
- **File**: `scenecheck-expo/app/create-event.web.tsx:35, 152-153, 751`
- **Category**: mock-leak
- **Description**: `SC_MY_ACCOUNTS` is imported from `@/data/mocks` at the top of the
  file without any mock-mode guard. In live mode the "Posting as" switcher and the
  `BasicsStep` account dropdown are entirely populated from this static fixture array
  rather than from the real session identity. The account name, handle, and id used to
  construct `previewEvent.hostId` and `previewEvent.host` are therefore always mock
  values in production.
- **Evidence**:
  ```ts
  // Line 35 — unconditional import
  import { SC_MY_ACCOUNTS } from '@/data/mocks';
  // Line 152-153 — used unconditionally in live mode
  const account =
    SC_MY_ACCOUNTS.find(a => a.id === activeAccount) || SC_MY_ACCOUNTS[0];
  ```
- **Mobile parity reference**: `scenecheck-expo/app/create-event.tsx` does not import
  `SC_MY_ACCOUNTS`; it derives the host from the session `me` object.
- **Spec reference**: AGENTS.md invariant — mock data imports must be inside a guarded
  `api.isMock()` branch.
- **Proposed fix**: Replace the `SC_MY_ACCOUNTS` reference with real store-derived
  account data. In live mode the "Posting as" switcher should read from
  `useStore(s => s.me)` and `useStore(s => s.orgPictures)` — the same slices already
  imported. Gate any fallback to `SC_MY_ACCOUNTS` under `if (api.isMock())`.
- **Reasoning**: A published event will have `hostId: 'me'` (a mock string) in the
  payload, causing broken navigation to `/profile/me` rather than the real UUID.

### F-R4-004: Boolean-precedence bug in draft step-restore
- **Severity**: medium
- **File**: `scenecheck-expo/app/create-event.web.tsx:334`
- **Category**: bug
- **Description**: The expression
  `initialDraft.lastStep ?? 0 ? Math.min(Math.max(initialDraft.lastStep, 1), 4) : 1`
  is parsed as `(initialDraft.lastStep ?? 0) ? (...) : 1`. The nullish-coalesce produces
  `0` only when `lastStep` is `null`/`undefined`; otherwise it produces the numeric
  `lastStep` value directly. Any truthy `lastStep` (including step 1 itself) takes the
  `Math.min/max` branch, so this part works. But if `lastStep` was saved as `0` — which
  `handleSaveDraft` passes as `step` when the user is on step 1 before ever moving —
  the expression evaluates to `1` correctly by accident. However if `lastStep` is
  `undefined` in a newly-created draft (no `lastStep` key), `?? 0` yields `0`, which is
  falsy, so the result is `1`. This is correct but only accidentally. More importantly the
  intent is unreadable and the intermediate `?? 0` is misleading — `0` is falsy so it
  triggers the `: 1` fallback anyway, making `?? 0` a no-op that just adds confusion.
- **Evidence**:
  ```ts
  setStep(initialDraft.lastStep ?? 0 ? Math.min(Math.max(initialDraft.lastStep, 1), 4) : 1);
  ```
- **Proposed fix**:
  ```ts
  setStep(
    initialDraft.lastStep != null
      ? Math.min(Math.max(initialDraft.lastStep, 1), 4)
      : 1
  );
  ```
- **Reasoning**: The current form is a precedence trap. A future reader trying to reason
  about it will likely misread `?? 0` as "default to first step if missing" — which is
  what the `?? 0` intended but doesn't actually achieve, because `0` is falsy. The clean
  null-check makes the intent explicit.

### F-R4-005: `visibility` field not passed to `api.createEvent` — the arch doc RPC
  omits it but the subscription flow needs it
- **Severity**: medium
- **File**: `scenecheck-expo/app/create-event.web.tsx:239-255`
- **Category**: wiring-gap
- **Description**: (See F-R4-001 for the primary finding.) A secondary consequence:
  the architecture doc's `create_event` RPC body does not include `visibility` in its
  listed parameters: `{title, description, start_at, end_at, location: {lat, lng},
  interests: [...], capacity}`. The web wizard adds `min_subscribers`,
  `location_name`, and price fields beyond that baseline — suggesting the Edge Function
  has already been extended — but `visibility` has apparently never been added. This
  means the gap likely exists in native too; neither client passes it. Flag for
  confirmation that the Edge Function's `CREATE TABLE` / INSERT handles a `visibility`
  column.
- **Spec reference**: FR5.1, Architecture §"User-created events"
- **Proposed fix**: Confirm the `create-event` Edge Function body includes a `visibility`
  parameter mapped to `events.visibility`; add it to both native and web callers.

### F-R4-006: `markNotificationRead` API call has no error path — error state mutates
  global loading slot
- **Severity**: medium
- **File**: `scenecheck-expo/hooks/useNotifications.ts:194-202`
- **Category**: bug
- **Description**: When `api.markNotificationRead` rejects, the catch block writes
  the error into `_initialLoading`... no wait — it writes to `_initialError`. That is
  correct in isolation. But `_initialError` is module-global and is also read as the
  `error` field in the hook's return value for ALL consumers (WebActivityPanel,
  WebNotificationsScreen, notifications.tsx). A transient read-error on a single
  mark-as-read PATCH will therefore make the notifications page render
  "Couldn't load activity right now" — the same error state as a full-fetch failure —
  even though data is still present and valid.
- **Evidence**:
  ```ts
  const markRead = useCallback(async (id: string) => {
    setCache(prev => prev.map(n => (n.id === id ? { ...n, read: true } : n)));
    if (mock) return;
    try {
      await api.markNotificationRead(id);
    } catch (e) {
      _initialError = e instanceof Error ? e : new Error(String(e)); // pollutes global error
      emit();
    }
  }, [mock]);
  ```
- **Proposed fix**: Give markRead errors their own module-level slot (e.g., `_markError`)
  and return it separately, or simply swallow mark-read errors silently (the optimistic
  flip is already in place; a failed PATCH is low-stakes):
  ```ts
  } catch {
    // Mark-as-read is best-effort; the optimistic flip stays.
  }
  ```
- **Reasoning**: Overwriting `_initialError` with a mark-read failure breaks the
  page-level error UX for all consumers simultaneously. The comment in the source already
  notes that "a stale-true is less surprising than a flicker back to bold" — the same
  principle applies to the error slot.

### F-R4-007: `waitlist_promotion` notification type has no dedicated row in
  `WebActivityRow` switch
- **Severity**: medium
- **File**: `scenecheck-expo/web/WebActivityRow.tsx:117-185`
- **Category**: wiring-gap
- **Description**: The architecture document lists `waitlist_promotion` as one of the
  push-notification trigger types (§"Push notifications — What triggers a push").
  The `notifications` Edge Function (dispatch) persists this as an in-app row of type
  `'waitlist_promotion'`. The `WebActivityRow` switch covers `friend_request`,
  `event_invite`, `event_reminder`, `event_update`, `chat_reply` / `chat_message`, and
  `rating_received`, but has no case for `waitlist_promotion`. The row falls through to
  the generic `default` branch which renders `payload.title ?? type.replace(/_/g, ' ')`
  — meaning it shows "waitlist promotion" as plain text with a generic bell icon and no
  event deep link. The same gap exists in the native `notifications.tsx:titleFor` switch.
- **Evidence**:
  ```ts
  switch (n.type) {
    case 'friend_request': ...
    case 'event_invite': ...
    case 'event_reminder': ...
    case 'event_update': ...
    case 'chat_reply':
    case 'chat_message': ...
    case 'rating_received': ...
    default: { ... } // waitlist_promotion falls here
  }
  ```
- **Spec reference**: Architecture §"Push notifications"
- **Proposed fix**: Add a dedicated case:
  ```ts
  case 'waitlist_promotion':
    return {
      icon: 'check-circle',
      accentToken: 'good',
      title: `You're in — ${eventTitle} has a spot for you`,
      body: preview || 'Tap to view event details.',
      cta: 'View event',
      ctaHref: eventId ? `/event/${eventId}` : deepLink,
    };
  ```
- **Reasoning**: A waitlist promotion is the highest-signal notification a user can
  receive — they've been waiting and a spot just opened. Rendering it as generic text
  with no action button is a significant UX regression.

### F-R4-009: `useNotifications` refcount can go negative if effect runs with
  `mock=true` on mount but `mock=false` on unmount (environment change between renders)
- **Severity**: low
- **File**: `scenecheck-expo/hooks/useNotifications.ts:169-182`
- **Category**: bug
- **Description**: The subscription effect guards its body with `if (mock) return;` —
  which means if `mock` is true at mount time, `_refCount` is never incremented, but
  the cleanup function returned by the effect still decrements `_refCount`. In a hot-
  reload or Strict Mode double-invoke scenario where `mock` switches between effect call
  and cleanup call, `_refCount` goes to `Math.max(0, -1) = 0`, which would prematurely
  call `closeSubscription()` even though another consumer is still mounted. In practice
  `api.isMock()` is stable (env vars don't change at runtime), so this is theoretical
  today, but the guard placement is fragile.
- **Evidence**:
  ```ts
  useEffect(() => {
    if (mock) return;         // ← early-return; no _refCount++ happens
    // ...
    _refCount += 1;
    // ...
    return () => {
      _refCount = Math.max(0, _refCount - 1);  // ← always runs, even if +1 never did
      if (_refCount === 0) closeSubscription();
    };
  }, [mock, meId]);
  ```
- **Proposed fix**: Move the refcount guard inside the cleanup, or capture whether the
  effect actually ran:
  ```ts
  useEffect(() => {
    if (mock || !isLiveBackendAvailable() || !supabase || !meId) return;
    // ... setup ...
    _refCount += 1;
    return () => {
      _refCount = Math.max(0, _refCount - 1);
      if (_refCount === 0) closeSubscription();
    };
  }, [mock, meId]);
  ```
  (The current code already has the `!isLiveBackendAvailable()` guard after the mock
  check, but the cleanup does not mirror it — the fix is to ensure the cleanup runs only
  when setup ran.)
- **Reasoning**: Safe today but will bite during testing or if the mock flag ever
  becomes dynamic (e.g., a runtime "force mock" toggle in tweaks).

## Cross-cutting concerns

### Notifications table existence

Migration **`00008_create_notifications.sql`** exists at
`supabase/migrations/00008_create_notifications.sql` and creates:

```sql
CREATE TABLE public.notifications (
  id           UUID PRIMARY KEY DEFAULT uuid_generate_v4(),
  user_id      UUID NOT NULL REFERENCES profiles(user_id) ON DELETE CASCADE,
  type         TEXT NOT NULL,
  payload_json JSONB DEFAULT '{}',
  read         BOOLEAN NOT NULL DEFAULT false,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

The schema exactly matches what `useNotifications` reads: `id`, `user_id`, `type`,
`payload_json`, `read`, `created_at`. The table is present in production. No missing-
table risk.

### `markAllRead` bypasses `api` abstraction

`markAllRead` in `useNotifications.ts` (lines 211-223) calls `supabase.from('notifications')`
directly rather than going through `api.markNotificationRead`. This is an intentional
batch optimization (documented in the comment), but it means `api.isMock()` is checked
separately from the `requireClient()` pattern used elsewhere, and any future RLS change
to the notifications table requires changes in two places. Low severity but worth noting
for maintainability.

### Calendar handoff placement (FR7.2)

The web wizard fires `googleCalendar.insertEvent` AFTER `api.createEvent` succeeds
(lines 258-277 of create-event.web.tsx). The architecture doc places this trigger at
event-SUBSCRIPTION time (FR7 §"Event subscription & calendar add"), not at creation
time. This is a deliberate platform deviation: the creator automatically subscribes to
their own event, so firing at creation-time is equivalent. The behavior is documented
in the comment `// FR7.2 fire-and-forget calendar insert` and matches the native
create-event.tsx. Not a bug, but auditors should note the intentional divergence from
the architecture table's step ordering.

## Healthy / well-implemented patterns

- **Module-singleton notifications hook**: The `useNotifications` refcount + `_listeners`
  fan-out pattern correctly prevents duplicate Supabase Realtime channels when multiple
  consumers mount simultaneously, and the user-switch path (`resetForUser`) cleanly
  tears down the old channel and resets the cache. This is non-trivial to get right and
  was done well.

- **Optimistic `markRead` with no rollback**: The comment explicitly defends the
  decision not to roll back the read-flag on failure ("a stale-true is less surprising
  than a flicker back to bold"). This is correct UX reasoning and the implementation
  follows through consistently.

- **Settings page store sync**: Every control in `settings.web.tsx` writes directly to
  the same Zustand slice actions (`setRadius`, `setVisibility`, `setNotifPref`,
  `setPalette`, `setMode`, `setTweak`, `setRailStyle`) used by the native settings page.
  There is no duplicated local state; the comment in the file header explicitly calls
  this out. This is the right pattern.

- **`WebActivityRow` friend-request buttons correctly use store + api**: The `accept`
  and `decline` handlers (lines 210-235 of WebActivityRow.tsx) follow the AGENTS.md
  optimistic-commit pattern — `acceptStore` / `declineStore` fire synchronously (the
  store update), then the API call happens async, with a rollback toast on failure. The
  same mutators used by the native friends screen are reused here — no parallel path.
